# Strategies for Accelerating FSTESTS Execution on User-Mode Linux (UML)

Running the `fstests` suite (formerly `xfstests`) on a full filesystem like BTRFS is notoriously time-consuming. While migrating to User-Mode Linux (UML) eliminates the overhead of full hardware virtualization (KVM/QEMU) and provides a fast boot time (<2 seconds), the execution of the tests themselves remains the primary bottleneck.

Based on our empirical timing runs, standard BTRFS tests average ~13.5 seconds per test on UML, with some stress tests taking upwards of 4 minutes each. Running the full suite of ~640 compatible tests sequentially takes over 2.5 hours.

To significantly speed up this feedback loop, we can employ several strategies tailored specifically to the UML environment.

## 1. Parallel Execution via Multiple UML Instances (The "Sharding" Approach)

The most significant advantage of UML over traditional VMs is that it runs as a standard Linux user-space process. This makes it trivial to spawn multiple isolated UML instances concurrently, each running a subset of the test suite.

### Implementation:
Instead of running `./check -g auto` in a single UML instance, you can:
1. **Partition the Test List:** Split the `uml_test_list.txt` into $N$ equal chunks (where $N$ is the number of available CPU cores on the host).
2. **Create Ephemeral Disk Images:** For each chunk, clone the base `rootfs.img` using Copy-on-Write (e.g., `cp --reflink=always` or `qemu-img create -f qcow2 -b`) and create temporary `test` and `scratch` images.
3. **Spawn Concurrent UML Processes:** Launch $N$ background UML processes simultaneously, passing a different chunk of tests to each via the kernel command line or a mounted configuration file.
4. **Aggregate Results:** Once all UML processes exit, merge their respective `results/check.log` files.

**Expected Speedup:** Near-linear scaling. Running 8 parallel UML instances on an 8-core machine can reduce a 2.5-hour run to under 25 minutes.

## 2. Tuning fstests Environment Variables (`TIME_FACTOR` and `LOAD_FACTOR`)

Many tests in `fstests` use internal looping mechanisms, primarily driven by the `fsstress` and `fsx` utilities, to simulate heavy I/O loads over time. The intensity and duration of these tests are controlled by environment variables defined in `common/config`.

By default, these variables are:
```bash
export TIME_FACTOR=1
export LOAD_FACTOR=1
```

### Implementation:
In your `local.config` or runner script, you can aggressively scale these down for CI/CD environments where rapid feedback is prioritized over deep soak testing:
```bash
export TIME_FACTOR=0    # Or a fractional value if supported, or 1 if 0 is invalid
export LOAD_FACTOR=0
export SOAK_DURATION=0
```
*Note: Some recent patches to `fstests` explicitly allow zero duration for `fsstress` and `fsx` to support rapid testing.*

**Expected Speedup:** Variable, but tests like `btrfs/004` (which normally takes ~4 minutes due to heavy `fsstress` usage) can be reduced to seconds.

## 3. Tiered Execution (Fail-Fast Strategy)

Not all tests provide the same return on investment regarding time spent versus bugs caught. The suite can be split into a "Quick" tier and a "Deep" tier.

### Implementation:
1. **Tier 1 (Smoke & Quick):** Run only tests tagged with `quick`. Our analysis shows there are ~516 quick tests (BTRFS + Generic) that take ~8-10 seconds each. Run these on every commit.
2. **Tier 2 (Stress & Soak):** Tests tagged with `stress`, or those that explicitly invoke `_run_fsstress` with high loop counts, are deferred to nightly runs. 

## 4. In-Memory Filesystems for UML Block Devices

Currently, the UML instances are backed by file-based block devices (`ubda=rootfs.img`, `ubdb=test.img`). This means all I/O generated by `fstests` inside UML is translated into host-level file I/O, which hits the host's page cache and eventually the host's physical disk.

### Implementation:
Move the backing files for the `TEST_DEV` and `SCRATCH_DEV` to a `tmpfs` (RAM disk) on the host system:
```bash
# On the host
mount -t tmpfs -o size=4G tmpfs /mnt/ramdisk
truncate -s 2G /mnt/ramdisk/test.img
truncate -s 2G /mnt/ramdisk/scratch.img
# Launch UML pointing to the ramdisk images
linux ubdb=/mnt/ramdisk/test.img ubdc=/mnt/ramdisk/scratch.img ...
```

**Expected Speedup:** Significant reduction in I/O wait times, especially for metadata-heavy BTRFS tests, as all operations occur purely in host memory.

## Summary Recommendation

To achieve the fastest possible `fstests` execution on UML:
1. Move all backing `.img` files to a host `tmpfs`.
2. Set `TIME_FACTOR=0` and `LOAD_FACTOR=0` in the UML `local.config`.
3. Implement a test sharding script that spawns $N$ parallel UML instances (where $N = \text{Host CPUs}$). 

Combining these three techniques can realistically reduce the total test execution time from **2.5 hours down to 5-10 minutes**, providing an incredibly fast feedback loop for BTRFS kernel development.
